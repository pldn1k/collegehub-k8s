Отлично! 🎉 Поздравляю с полноценно работающим кластером.

Теперь — разбор полётов: почему всё работало через kubectl port-forward, а через Ingress/браузер нет, и что было самым частым камнем преткновения.

Ниже — реальный боевой опыт, оформленный без воды. Чётко, по делу, без угадаек.

1. Главная причина всех проблем — CORS
Вылезла именно при доступе из браузера, потому что фронтенд и API крутятся на разных портах (localhost:30080 vs localhost:30081).
Через kubectl port-forward это не проверяется.

Почему ломало:
Браузер шлёт preflight OPTIONS‑запрос перед POST

API‑шлюз должен вернуть специальные заголовки (Access-Control-Allow-Origin и т.д.)

Если их нет — браузер молча блокирует запрос и не показывает логин

Что решило:
✅ Глобальный обработчик OPTIONS в api-gateway

✅ @app.after_request для добавления CORS‑заголовков к каждому ответу

✅ Явный ответ 200 на OPTIONS без проверки авторизации

2. Сети внутри Kubernetes — это не localhost
Локально команда работала, а внутри кластера — нет, потому что использовались неправильные имена или порты.

Типичные грабли:
auth-service:6382 – если Redis слушает на 6379

postgres:5432 – если настоящий порт 5432, но где‑то указан другой

Переменные окружения не проброшены в под

Решение:
✅ В кластере всегда используй DNS‑имена сервисов (postgres, redis, auth-service)

✅ Порты должны быть одинаковыми в Dockerfile, манифестах и коде

3. Образы не обновляются в Kind
Ты пересобираешь docker build, а в кластере всё ещё старый код.
Потому что Kind кэширует образы, если не загрузить их заново.

Как это лечится:
bash
kind load docker-image api-gateway:latest
kubectl rollout restart deployment api-gateway
Без этих команд изменения в образе не попадут в кластер.

4. Ingress и MetalLB — сложная связка для локальной разработки
Ingress хорош для продакшена, но для отладки он добавляет:

сложный синтаксис

ещё один слой сетевых маршрутов

дополнительные ошибки из‑за несовпадения IP и портов

Что было проще:
✅ NodePort для фронтенда (30080) и для API (30081)
✅ Напрямую обращаться к портам хоста → меньше магии, проще понять, где сбой

5. Helm — удобно, но требует полного понимания манифестов
Helm хорош, когда приложение уже стабильно работает.
Пока идут активные правки, он создаёт лишний уровень абстракции.

Что мешало:
шаблонизация {{ .Values.namespace }} давала странные ошибки, если поле не определено

Helm‑релиз мог оставаться в failed состоянии

helm uninstall не удалял PVC и PV

Вывод:
Helm — для упаковки готового приложения.
Пока код меняется, проще использовать kubectl apply -f manifests/

6. База данных PostgreSQL и PV в Kind
Kind — это кластер в контейнере. Если пересоздать кластер — данные исчезнут.

Что работало:
hostPath + ручное создание PV

SQL‑скрипт инициализации через init.sql

Что не работало без явного указания:
автоматическое создание PV

автоподключение PVC

7. Итог: алгоритм, который спас бы недели экспериментов
Проверить руками через curl

bash
curl -X POST http://localhost:30081/api/auth/login -d '{"username":"student1","password":"pass123"}'
Посмотреть CORS в браузере (F12 → Network)

есть ли OPTIONS‑запрос?

приходят ли заголовки Access-Control-… ?

Сверить порты внутри кластера и в коде

Перезагрузить образ в Kind после сборки

На время отладки использовать NodePort вместо Ingress

Frontend на порту 30080

API Gateway на порту 30081

PostgreSQL, Redis, Auth‑сервис работают внутри кластера

CORS полностью настроен

Логин через браузер работает без дополнительных телодвижений

Приложение управляется через kubectl
